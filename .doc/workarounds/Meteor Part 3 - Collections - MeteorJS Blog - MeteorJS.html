Meteor Part 3 - Collections

Peter Ilfrich 270007T4VN | Mar 20 2015 | Tags:  mongodb publication javascript subscription collections schema meteor ‎ | 4,220 Views
0 people like this
 
MeteorJS (http://meteor.com) is a Javascript server/client framework based on an Express server.

This article is part of a blog series:

  1. Part 1 - Introduction
  2. Part 2 - Getting Started
  3. Part 3 - Collections
  4. Part 4 - Router
  5. Part 5 - Templates
  6. Part 6 - Deployment & BlueMix
  7. Part 7 - Automated Testing
MongoDB collections are the primary persistent storage for business objects in Meteor. In theory you can dump any JSON structure into a collection and MongoDB will eat it and spit it out again on request. When a document is inserted into MongoDB it receives an internal unique object ID, which can be used to identify the document.
 
A Meteor project should contain a "models" folder. This is a good place to store you collection declarations (all in one place). For the purpose of this article imagine a collection of regions, which are described by a mandatory name, an optional short handle and a flag declaring them as public regions or private regions:
 
regions = new Mongo.Collection('regions');
 
Using the package aldeed:collection2 will allow you to specify an optional schema for each collection. Although this might prevent inserts from time to time (because some of the schema constraints are not met), specifying a schema for a collection will make your application more robust in the long run. For a prototype or demo, you can ignore the schema declaration and rely on your testing to make sure the object structure inserted is correct. Long-term I recommend using schema declarations, which should be specified in the same file as the collection declaration.
 
regions.attachSchema(
    new SimpleSchema({
        name: {
            type: String
        },
        shortHandle: {
            type: String,
            optional: true
        },
        public: {
            type: Boolean
        }
    })
);
 
Just declaring the collection doesn't allow you to access it on the client-side though. To allow client-side access Meteor introduces publications on the server-side (usually inside a file in the server/startup folder):
 
Meteor.publish('regions', function () {
    return regions.find();
});
 
In this simple example, we're publishing all records. It is possible to reduce visibility to a sub-set of documents (e.g. depending on the logged in user). Basically you can use any logic inside the publication declaration:
 
Meteor.publish('regions', function() {
    if (Meteor.userId) {
        return regions.find();
    }
    else {
        return regions.find({ public: true });
    }
});
 
To complete the model declaration you need a subscription on the client-side as well (usually inside a file in the client/startup folder):
 
Meteor.subscribe('regions');
 
Now the client knows an object called "regions", which allows find(..), insert(..), update(..) and remove(..) operations on the entire collection. Be careful not to re-declare any variable name you are using for collections. 
 
Now this might seem like a security disaster. Basically anyone could just open the web site and read, modify, insert or delete any item in the collection without even having to log in. For a prototype, demo or PoC this is perfectly fine and speeds up development. If you want to go into production, you need to secure the data.
 
There are means to restrict data access from the client and the server by adding permission directives to your data model. These directives are defined in the following way:
 
if (Meteor.isClient) {
    regions.allow({
        insert : function () {
            return false;
        },
        update : function () {
            return false;
        },
        remove : function () {
            return false;
        }
    });
}
 
This would restrict the client to only read regions, buy you cannot directly insert/update/remove regions from the client-side. Similar code can be written for Meteor.isServer, where we define the access to that collection from the server side. Usually you would allow any manipulations, but if you have immutable records, you might set the permission for update and remove operations to false (prohibited). These directives also allow you to include logic and access to context variables (session, current user, etc).
 
Tipps, Tricks and Potholes
 
For the special case of accessing a specific object from the database (by using it's internal unique ID) you would normally run a query like this:
 
regions.findOne({ _id: '6LdDpLPKkigXMpFdp' });
 
As an alternative you can simply provide the ID as a single String parameter (short notation):
 
regions.findOne('6LdDpLPKkigXMpFdp');
 
This will return the JSON document directly. The findOne operation is special. You would expect a regular find(..) operation to return an array of JSON documents from the database. However what you really get is a LocalCollection.Cursor object. To retrieve the elements represented by this cursor you would use the fetch() operation, which returns the array of documents:
 
regions.find({public: true}).fetch();
 
During development there was another issue that came up and was hard to detect or debug: If you're using $pull and $push operators in one single query, you cannot modify the same field. Assume the case where your document contains an array of values and you want to replace one value by another value. What you would do is $pull the entry you want to replace and $push the new entry into the array. If you do that in one single statement, you will not succeed. MongoDB doesn't allow that. Instead you need to run 2 separate update statements: one to $pull and one to $push.
